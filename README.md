# Totally Normal Text Editor

A totally normal text editor. You type, and when you finish a word, that word
decides how it feels about being typed: "gravity" falls, "fire" flickers,
"rainbow" gets a hue per letter. Each completed word is sent to Claude, which
writes a small pure-math `simulate()` function describing that word's motion
and color; the word then breaks free of the document and animates across the
viewport.

Developed by Astraea.

## Run locally

```bash
npm install
cp .env.example .env   # then put your Anthropic API key in .env
npm start              # http://localhost:3000
```

Requires Node 18+ (uses the built-in `fetch`).

## Deploy

Any Node host works (Fly, Railway, Render, a VPS behind nginx...). The only
configuration is environment variables:

- `ANTHROPIC_API_KEY` ‚Äî required, server-side only. It is never sent to the
  client; the frontend only ever talks to this app's own `/api/simulate`. Acts
  as the shared "host" key (see limits below).
- `HOST_WORD_LIMIT` ‚Äî optional (default `10`). How many brand-new words one
  visitor may generate on the host key before being asked to bring their own.
- `TRUST_PROXY` ‚Äî set when behind a reverse proxy so per-IP limits see the real
  client IP (e.g. `1`). Leave unset locally.
- `PORT` ‚Äî optional, defaults to 3000.

## Host limit & bring-your-own-key

Each visitor may generate up to `HOST_WORD_LIMIT` brand-new words on the shared
host key. **Cached words never count** ‚Äî once any word is in the shared store,
everyone gets it for free forever, so the limit only meters genuinely new
generations. When a visitor exceeds it, an uncached word is locked in as red
plain text and a popup invites them to paste their own Anthropic key. That key
is stored only in their browser (`localStorage`), sent with subsequent requests,
used for their generations (bypassing the host limit), and **never stored or
logged server-side**. Words they generate still land in the shared store, so
everyone benefits. A rejected key produces a distinct `bad_key` response so the
client can prompt them to re-enter it.

The counter and the per-IP burst guard are in-memory (per instance, reset on
restart) ‚Äî move them to Redis alongside the store for a multi-instance site.

## The shared word store ("generate each word once, for everyone")

Every word is generated by Claude exactly once. The first visitor to type a
given word triggers the API call; the resulting `simulate()` source is written
to a durable shared store, and every later request ‚Äî from any visitor, and
across server restarts ‚Äî is served from the store instantly with no API call
(same idea as Infinite Craft's shared recipe cache).

The store lives in `store.js`, backed by a single JSON file at `data/words.json`
(created on first write, gitignored). It is loaded fully into memory on startup
and rewritten atomically only when a genuinely new word is added, so cache
hits cost nothing and a crash can't corrupt it. An in-flight map in `server.js`
also collapses simultaneous first-time requests for the same new word into a
single generation.

**Scaling to a public multi-instance site.** The JSON file is correct for a
single server instance (which is all a server-side cache needs to be shared
across all users). To run several instances or serverless functions behind a
load balancer, replace `store.js` with a Postgres/Redis/SQLite-backed module
exposing the same async `get(word)` / `set(word, code)` / `size` surface ‚Äî
nothing in `server.js` changes. Do the same for the currently in-memory,
per-instance rate limiter (move it to Redis).

## Words that don't animate

Not every word should come alive. Three kinds resolve to a shared no-op ‚Äî the
word just stays as plain typed text (`/api/simulate` returns `{ code: null }`):

1. **Common function words** ‚Äî articles, pronouns, prepositions, conjunctions,
   auxiliaries, filler ("the", "and", "this", "like", ...). The list lives in
   `words.js`; edit `STOPWORDS` to taste. These never reach Claude.
2. **Obvious keyboard-mash** ‚Äî caught by a cheap, conservative heuristic in
   `words.js` (`looksLikeGibberish`: no vowels, or a 6+ consonant run). Also
   never reaches Claude.
3. **Pronounceable non-words** ‚Äî e.g. "aduvnirfudjaifj", which no heuristic can
   safely distinguish from a real word. Claude itself judges these: the system
   prompt tells it to reply with the token `UNKNOWN` for unrecognized words.
   A refusal is cached as a tiny `__unknown__` marker (see `NOOP` in
   `server.js`), NOT as a generated function ‚Äî so an unrecognized word costs
   exactly one Claude call ever, and `data/words.json` never fills up with
   junk functions.

Stopwords and heuristic-gibberish are deterministic, so they cost no API call,
no storage, and don't count against the rate limit.

## Security model (read before "simplifying")

Generated code is untrusted, always. The **load-bearing boundary is the Web
Worker sandbox** in `public/worker.js`: the AI-written `simulate()` function
is compiled and executed only inside a Worker ‚Äî a separate JS realm with no
DOM ‚Äî whose network/storage capabilities (`fetch`, `XMLHttpRequest`,
`WebSocket`, `importScripts`, `indexedDB`, ...) are additionally stripped at
startup, and whose compilation scope shadows those names again. The worker
returns nothing but sanitized plain numbers (plus a regex-validated `#rrggbb`
string) per letter per frame; the main thread applies those as CSS transforms
to elements it already owns. The forbidden-token regex in `server.js` is
**defense in depth, not the boundary** ‚Äî it cheaply rejects obviously bad
output before caching, but passing it proves nothing. Likewise the shared
word‚Üícode store (`data/words.json`) is safe to share across visitors ‚Äî and
safe to persist to disk ‚Äî *only because* every client executes the stored
code inside its own Worker sandbox; the store holds already-checked source and
is not itself a trust boundary. Do not give
`simulate()` DOM access to unlock fancier effects; everything needed is
expressible as numbers per frame ‚Äî per-letter x/y/rotation/scale/skew/opacity/
color/glow, a capped list of stateless decorative particles, and a capped list
of extra single-character text glyphs (clones, appended letters, spawned
words), all sanitized by the worker before they cross back to the main thread.
